# Design: registry-tasks

## Approach

先为任务注册表定义一个独立规格面，明确最小行为边界，再由实现阶段提供一个默认的内存版本。

本提案将任务注册表限制为“按 ID 管理单个任务”的能力集合，不引入任务列表、搜索、持久化或跨进程共享语义。这样可以先把任务对象和状态机约束定稳，再让后续变更在此基础上扩展更高层行为。

任务对象以最小字段集为中心：
- `id` 作为调用方提供的稳定标识
- `title` 作为人类可读描述
- `status` 体现生命周期阶段
- `created_at` / `updated_at` 体现时间语义
- `metadata` 作为可选扩展字段承载附加信息

状态流转作为该变更的核心约束，由规格显式列出合法路径，并要求非法流转失败且不改变既有任务状态。

## Key Decisions

- v1 不包含 `List` 能力，避免在首个注册表提案中连带引入排序、过滤、分页和 deleted 可见性等额外决策
- 任务 ID 由调用方提供，不在注册表层引入 ID 生成策略
- 删除采用状态变更到 `deleted` 的语义，而不是物理移除，以保持生命周期可观察性
- `completed` 和 `deleted` 视为终态，避免后续行为出现回滚歧义
- 新规格聚焦任务注册表本身，不与团队、cron 或公开传输层一起打包推进

## Alternatives Considered

- 在同一变更中同时交付 `registry-tasks` 和 `registry-teams`
  - 未采用，因为会扩大 spec 面并混合两类对象的边界决策
- 在 v1 中直接加入 `List`
  - 未采用，因为这会迫使提案同时回答集合顺序、筛选范围和终态可见性
- 自动生成任务 ID
  - 未采用，因为这会把资源创建策略提前固化到基础注册表层
- 支持更宽的状态模型，例如 `blocked`、`failed`、`archived`
  - 未采用，因为当前 roadmap 只要求最小状态流转闭环
